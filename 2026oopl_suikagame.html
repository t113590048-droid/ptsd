專案簡介

遊戲簡介

  • 在 OOPL 課程初期，考量到開發時程與團隊能力，我們希望能選擇一款機制簡單耐玩、適合大眾且能完整復刻體驗的遊戲，因此看上了這款許多遊戲主播在閒聊時會搭配遊玩的熱門作品。我們希望透過這次專案，挑戰物理引擎與即時動畫的深度結合。
  • 本專案不僅還原了經典合成機制，還透過 Box2D 物理引擎模擬真實的水果碰撞。為了提升互動性，我們加入了獨特的 NPC 雲朵系統、自定義開場門簾動畫與 NPC 表情變化，使遊戲內容更具完整性與生命力。

組別分工

  • 本專案為兩人合作開發，架構設計、物理邏輯、UI 動畫、素材整合與除錯工作皆由兩位組員分階段協力完成，以確保遊戲體驗的順暢與程式碼的可維護性。
  • 開發過程：
    • 初期： 劉文軒主要負責遊戲主體架構規劃；黃楷程負責部分角色程式設計、模擬 Switch 原版遊戲環境進行素材拆包與可行性分析，確保美術風格的一致性。期中報告時因進度規劃曾有分工不均的情況。
    • 期中後： 轉由黃楷程主導後期的程式架構優化與新關卡設計（含障礙物、傳送門邏輯）；劉文軒則持續協助進行創意發想、機制測試並確保內容連貫性，最後由黃楷程統整最終報告與展示影片。

遊戲介紹

遊戲規則

  • 操作方式
    • A / D - 移動雲朵（控制水果掉落位置）
    • Space - 丟下當前水果
    • P - 遊戲暫停
    • R - 遊戲結束後重試 / 回到選單
    • C - 開啟作弊模式 (Debug)
  • 合成機制
    • 兩個相同種類的水果接觸後，會合成更高級別的水果，並依照等級給予相應分數。
    • 水果會受重力影響，在容器中進行物理碰撞與擠壓。
  • 死亡判定與動畫
    • 若水果堆疊超過頂部限制線，觸發死亡倒數機制：
      • 5 秒緩衝期： 正常遊戲顯示，供玩家做最後挽救。
      • 3 秒哭鬧震動期： 觸發 IsInCryStage，水果變成哭臉並隨機震動，展現遊戲結束前的掙扎。
      • Game Over： 跳轉至最終結算畫面，顯示分數。
  • 特殊功能
    • NPC 雲朵： 跟隨掉落軌跡移動，並具備隨機眨眼功能。
    • 開場動畫： 遊戲開始時有門簾展開與氣泡散開動畫。
    • 額外關卡： 包含傳送門 (Portal)、移動平台 (Moving Platform) 及靜態障礙物等多種擴充玩法。

程式技術

  • 素材分析與資源模擬 (Asset Analysis)
    • Switch 環境模擬： 透過分析 Switch 原版西瓜遊戲的資源結構進行拆包與分析，以此為基準模擬原版的物理參數（如摩擦力、回彈係數），確保手感的一致性。
    • 資源重製： 將解析後的素材進行格式轉換與整合，使其完美對應 SDL2 的渲染系統，並確保與我們自定義 UI 的風格融合。
  • 物理模擬 (Box2D)
    • 使用 PTM (Pixel-to-Meter) 轉換比率，將像素座標轉換為 Box2D 物理單位，處理精準的碰撞與合成邏輯。
  • 狀態機管理
    • 使用 IsGameOver 與 IsInCryStage 狀態機，精準控制遊戲從正常遊玩、瀕死倒數到死亡前夕的動畫轉換。
  • 關卡邏輯 (Extra Features)
    • 實作了「傳送門」物件，透過檢測水果座標並傳送至對應位置；以及「移動平台」機制，豐富了遊戲的多樣性。
  • 作弊模式 (Cheat Mode)
    • 加入了 C 鍵偵測，允許開發者在測試與評分時跳過部分限制，加速遊戲流程與機制驗證。

結語

問題與解決方法

  • 狀態衝突與動畫卡頓：

    在實作死亡動畫時，最初遇到「死亡瞬間水果仍然震動」或「哭鬧動畫無法觸發」的問題。解決方法是將 Update 中的 IsInCryStage 判定邏輯移至繪製迴圈的最前端，並明確區分「CryStage 繪製」與「GameOver 結算」的狀態。

  • 亂數產生器規律化：

    發現水果生成亂數在不同次執行下過於規律，解決方法是將亂數種子設定移動至程式啟動的第一階段（Initialization），確保每次遊玩體驗的完全隨機性。
